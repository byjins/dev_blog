> 뭐.. 해봤자 Semantic Tag를 사용하고 Img태그에 alt를 넣는 정도..?가 끝이었다.

> 요즘 앱들을 참 화려하고 애니메이션이 가득한 화면이 많은 것 같다. 하지만 애니메이션이 가득한 이쁘고 멋진 프로젝트를 만들면 끝일까?

프론트엔드 개발자라면 장애가 있는 사람을 포함한 모든 사용자가 동등하게 정보와 서비스를 이용할 수 있도록 UX/UI 적으로 많은 걸 보장해줄 수 있어야 한다고 생각한다.

지금부터 웹접근성에 대해 알아보고 어떻게 적용할 수 있는지 알아보자!

## Web 접근성이란?
Web 접근성`(Web Accessibility, a11y)`이란 장애가 있는 사람을 포함한 모든 사용자가 웹사이트나 애플리케이션을 문제없이 이용할 수 있도록 보장하는 것을 의미한다.
`W3C(World Wide Web Consortium)`에서 제공하는 `WCAG(Web Content Accessibility Guidelines)`가 대표적인 가이드라인이다.

## Web 접근성이 왜 필요할까?
1. `법적/윤리적 이유`
    - 장애인 차별 금지법(ADA, 미국), 웹 접근성 지침(KWCAG, 한국) 등 다양한 법률에서 웹 접근성을 요구합니다.
    - 모든 사람이 동등하게 정보와 서비스를 이용할 수 있도록 해야 합니다.

2. `사용자 경험 개선`
    - 장애가 없는 사용자도 웹 접근성이 좋은 사이트에서 더 편리하게 서비스를 이용할 수 있습니다.
    - 예: `대체 텍스트(alt text)`가 잘 제공되면 네트워크 속도가 느릴 때 이미지가 로딩되지 않아도 내용을 파악할 수 있습니다.

3. `SEO(검색 엔진 최적화)에 유리`
    - 접근성이 높은 사이트는 검색 엔진 크롤러가 쉽게 분석할 수 있기 때문에 SEO 점수가 올라가는 효과가 있습니다.

4. `더 넓은 사용자층 확보`
    - 장애인뿐만 아니라 고령층, 저사양 기기 사용자, 화면이 작은 모바일 사용자를 포함해 더 많은 사람들에게 서비스를 제공할 수 있습니다.

## Web 접근성을 어떻게 적용할까?
WCAG에서는 4가지의 원칙을 기준으로 접근성을 평가한다고 합니다.
1. `Perceivable(인식 가능성)`
2. `Operable(운용 가능성)`
3. `Understandable(이해 가능성)`
4. `Robust(견고성)`

### Perceivable(인식 가능성)
사용자가 콘텐츠를 `보고`, `듣고`, `읽을` 수 있어야합니다.

> ✅ 텍스트 기반 대체 제공
- alt 속성을 이용해 `이미지의 대체 텍스트`를 제공하기
```html
<img src="chart.png" alt="2024년 매출 그래프 - 1월부터 12월까지의 증가 추세">
```
- 의미 없는 장식용 이미지에는 alt=""를 제공해 스크린리더가 무시하도록 설정해야합니다.
```html
<img src="decorative.png" alt="">
```


> ✅ 영상 및 오디오에 자막/대체 텍스트 제공
- 동영상에는 `자막을 추가`하거나, `스크립트를 제공`해야합니다.
```html
<video controls>
  <source src="video.mp4" type="video/mp4">
  <track src="subtitles.vtt" kind="subtitles" srclang="ko" label="Korean">
</video>
```

> ✅ 색에 의존하지 않는 정보 제공
- 색깔만으로 정보를 전달하지 않도록 해야합니다.(예: 빨간색 오류 메시지 대신 텍스트 추가)
```html
<p style="color: red;">⚠ 비밀번호를 입력해주세요.</p>
```

### Operable(운용 가능성)
사용자가 `키보드`, `마우스`, `스크린 리더` 등 다양한 방식으로 사이트를 조작할 수 있어야 합니다.

> ✅ 키보드만으로 모든 기능 조작 가능
- 버튼, 링크, 폼 요소는 Tab 키로 이동이 가능해야 합니다.
- aria-label이나 aria-labelledby를 사용해 스크린 리더가 읽을 수 있도록 만들어야합니다.

aria-label
```html
<button aria-label="검색 버튼">🔍</button>

➡ 스크린 리더는 “검색 버튼“이라고 읽는다.
```
aria-labelledby
```html
<h2 id="formTitle">회원가입</h2>
<p id="formDesc">이 버튼을 누르면 등록됩니다.</p>
<button aria-labelledby="formTitle formDesc">✅</button>

➡ 스크린 리더는 “회원가입, 이 버튼을 누르면 등록됩니다.“라고 읽는다.
```

> ✅ 자동 재생 콘텐츠 피하기
- 자동으로 재생되는 배경음악, 동영상은 `사용자가 멈출 수 있어야 합니다`.
```html
<video autoplay muted loop>
  <source src="background.mp4" type="video/mp4">
</video>
```

> ✅ 초점이 보이도록 스타일 추가
- 사용자가 현재 어디에 있는지 알 수 있도록 스타일을 명확히 설정해야합니다.
```html
button:focus {
  outline: 2px solid blue;
}
```

### Understandable(이해 가능성)
사용자가 `웹사이트를 쉽게 이해하고 사용`할 수 있어야 합니다.

> ✅ 간결한 문장과 쉬운 용어 사용
- 복잡한 전문 용어를 줄이고, 쉬운 설명 제공하기
- 긴 문장은 리스트나 섹션으로 나누기

> ✅ 폼 요소에 레이블 제공
- 입력 필드에는 label을 명확히 연결해야합니다.
```html
<label for="email">이메일:</label>
<input type="email" id="email" name="email">
```

> ✅ 에러 메시지는 친절하고 구체적으로 제공
- 어떤 문제인지, 어떻게 해결해야 하는지 설명해야 합니다.
```html
<p style="color: red;">⚠ 비밀번호는 최소 8자 이상이어야 합니다.</p>
```

### Robust(견고성)
웹사이트가 다양한 기술(스크린 리더, 보조 기기, 최신 브라우저)에서도 정상적으로 작동해야 합니다.

> ✅ 올바른 HTML 사용
- role, aria- 속성을 활용해 웹 요소의 역할을 명확히 해야합니다.
```html
<nav role="navigation">
  <ul>
    <li><a href="#">홈</a></li>
    <li><a href="#">서비스</a></li>
  </ul>
</nav>
```

> ✅ 반응형 디자인 적용
- 모바일, 태블릿, 데스크톱에서 `일관되게 작동하는 레이아웃` 사용하기
```html
@media (max-width: 768px) {
  body {
    font-size: 14px;
  }
}
```

> ✅ 보조 기술(스크린 리더, 음성 명령 등)에서 테스트
- NVDA, VoiceOver, JAWS 같은 스크린 리더로 웹사이트 테스트하기

## Web 접근성 검사 및 도구
접근성을 검사할 수 있는 유용한 도구들
1. Lighthouse(chrome DevTools)
    - Chrome 브라우저에서 F12 → “Lighthouse” → “Accessibility” 체크
2. axe DevTools(확장 프로그램)
    - 크롬 익스텐션 설치 후 웹페이지의 접근성 오류 검사
3. WAVE(Web Accessibility Evaluation Tool)
    - 웹사이트를 입력하면 접근성 문제를 시각적으로 분석해줌
    - https://wave.webaim.org/
4. NVDA: NVDA(NonVisual Desktop Access)
    - 무료 오픈소스 스크린리더
    - https://www.nvaccess.org/
